# N1Hub Hot Roadmap Q2 2026

Updated: 2026-03-09

This roadmap is the wave-based delivery plan behind the hot queue. It is not a fantasy roadmap. It only includes work that is already visible in the current repository.

## Wave 1: Branch Reality First

Primary target:

- freeze the Real/Dream field into a deterministic audit
- triage the highest-drift constitutional hubs
- resolve the dream-only Vault Steward outliers
- reconcile Real-first engineering law with Dream branch policy

Expected outcomes:

- the current Real/Dream corpus is measured, ranked, and no longer hand-wavy
- top constitutional hub overlays have explicit promote, retain, or rewrite decisions
- dream-only Vault Steward operational capsules have lifecycle law
- real-only engineering doctrine stops living outside branch policy

Blocking risks:

- agents touching Dream without a shared drift picture
- constitutional hubs remaining in permanent speculative limbo
- operational Dream capsules remaining undocumented runtime residue

Exit criteria:

- one shared Real/Dream field brief exists and is trusted
- the named constitutional hubs all have explicit branch decisions
- the three dream-only Vault Steward capsules have lifecycle law
- the nine real-only engineering law capsules have explicit branch-policy treatment

## Wave 2: Runtime Hardening After Branch Triage

Primary target:

- make A2C query and daemon semantics safe
- reduce the highest-risk seams in the active Vault Steward cluster
- complete A2C wave 2 contract tests
- expand Real/Dream promotion and merge tests
- harden validator/API boundary package
- split Symphony contract layers

Expected outcomes:

- A2C query defaults are read-only
- Vault Steward gets one more bounded extraction step with tests
- A2C runtime has real contract coverage across recon, query, watch, autonomous, and ingest
- Real/Dream promotion paths have stronger API and merge evidence
- validator CLI, API, and OpenAPI surfaces stay in parity
- Symphony law, prompts, and tracker state are less entangled

Blocking risks:

- promotion logic staying under-tested while corpus decisions accelerate
- documentation outrunning code reality
- OpenAPI drift
- large-file pressure increasing during refactors

Exit criteria:

- A2C query semantics are explicit and test-backed
- the next Vault Steward extraction step is bounded and verified
- Real/Dream promotion routes have stronger proof
- validator and Symphony surfaces are still explicit after hardening work

## Wave 3: Capsule-Native Planning and CI Pressure

Primary target:

- enforce changed-files file guardrails in CI
- stabilize the hot queue as a standing execution surface
- design the capsule-native planning and delegation control plane behind the hot queue
- introduce cross-model review and recurring N1 iteration as bounded execution lanes
- open the A2C path from raw operator input into task packets and future task capsules

Expected outcomes:

- changed-files gate catches new large-file regressions without exploding on historical debt
- `TO-DO/` becomes the default agent intake surface for hot work
- roadmap, goal, milestone, and task capsules have a clear path to become the durable execution substrate
- the engineering pipeline gains a hostile second-pass review lane before wider autonomy ramps further
- N1 iteration becomes scheduler-friendly without collapsing into an unsafe hidden daemon
- user input can move through A2C into bounded N1Hub work instead of dying as chat context

Blocking risks:

- trying to enforce full-history file guardrails before the repo is ready
- stale queue items drifting away from actual implementation state
- task planning remaining trapped in markdown after the vault already exposes stronger planning primitives
- review automation pretending to be cross-model while still using same-model subagents
- recurring iteration loops mutating planning state noisily without meaningfully advancing execution
- user-intent ingestion staying informal, so operator requests still have to be manually translated into queue state

Exit criteria:

- changed-files guardrail policy is explicit
- the queue acts like a real intake surface instead of a note dump
- operator input has an intake contract and packet-builder path
- scheduled N1 iteration stays one-pass and proof-bearing

## Wave 4: Context and Memory Topology

Primary target:

- formalize hot, warm, and cold context layers for N1Hub
- connect root docs, memory surfaces, skills, and capsule-native planning into one routing model

Expected outcomes:

- agents load the right context layer for the right kind of work
- memory surfaces stop drifting into overlapping pseudo-documentation
- future retrieval or compressed repo-map work has a clear architectural home

Blocking risks:

- adding fashionable context files without actual routing value
- letting external agent playbooks outrun repo-native adaptation discipline

Exit criteria:

- hot, warm, and cold context layers are explicit
- root docs, memory surfaces, skills, and queue law stop overlapping ambiguously
- future retrieval or repo-map work has a clear home instead of becoming another floating note

## Success Criteria

By the end of this roadmap wave, N1Hub should have:

- a measured and triaged Real/Dream field instead of branch folklore
- explicit treatment of dream-only operational capsules and real-only engineering law
- safer A2C operational semantics
- lower blast radius in the top active cluster
- stronger contract tests on real runtime boundaries
- better CI pressure against new architecture debt
- a living hot queue that future agents can follow without rediscovery
- a repo-native path from operator input to A2C intake to executable task packets
