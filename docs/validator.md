# Capsule Validator

## Purpose
The Capsule Validator is the integrity guardrail for N1Hub and CapsuleOS. It enforces the 5-Element Law and the 16 Validation Gates across API routes, UI editing flows, import/A2C ingest paths, CLI usage, CI, and scheduled audits.

## Architecture
The core engine lives in `lib/validator/` and is shared by all integration surfaces.

- `core.ts`: `CapsuleValidator` class, gate orchestration, dependency injection, cache.
- `schemas.ts`: Zod schemas for root structure, confidence vectors, and field constraints.
- `gates.ts`: Gate metadata (`G01` to `G16`) and constants.
- `autofix.ts`: Safe auto-fixes (G10, G11, G15 + optional seal recompute).
- `utils.ts`: token/word counting, semantic hash checks, confidence normalization, integrity hash.
- `types.ts`: API contracts and interfaces.
- `index.ts`: public exports and functional API wrappers.

## Public API

```ts
import { validateCapsule, autoFixCapsule, createCapsuleValidator } from '@/lib/validator';

const result = await validateCapsule(capsule, {
  existingIds: new Set(['capsule.foundation.capsuleos.v1']),
  skipG16: false,
});

const { fixedData, appliedFixes } = autoFixCapsule(capsule);
```

## Validation Gates

| Gate | Check | Auto-fix |
|---|---|---|
| G01 | Exactly 5 root keys | No |
| G02 | Required subfields present | No |
| G03 | Type/enum/version correctness | No |
| G04 | `provenance_coverage >= 0.85`; source required when `< 1.0` | No |
| G05 | Preferred `content_type` (`markdown/json/text`) | Warning |
| G06 | Payload integrity (`truncation_note` string if present) | No |
| G07 | Summary 70-160 words | No |
| G08 | Keywords count 5-15 | No |
| G09 | Semantic hash format | No |
| G10 | Semantic hash parity between metadata and neuro layer | Yes |
| G11 | Canonical relation types; `refutes` deprecated | Yes |
| G12 | Link targets exist in known ID set | No |
| G13 | Non-draft capsules must have at least one link | No |
| G14 | Coherence trap for long payloads vs contradiction pressure | No |
| G15 | Confidence vector shape/range | Yes |
| G16 | Integrity seal (JCS + SHA3-512) | Optional recompute |

## Auto-fix Behavior
Auto-fix is intentionally non-destructive.

- G10: Sync semantic hash parity when one side is valid and the other is not.
- G11: Replace `refutes` with `contradicts`.
- G15: Normalize confidence vector array shape to canonical object shape.
- G16: Optionally recompute `integrity_sha3_512` after safe format fixes.

## API Endpoints

- `POST /api/validate`: validate single capsule.
- `POST /api/validate/batch`: validate array of capsules (owner role).
- `POST /api/validate/fix`: auto-fix capsule and return corrected payload (owner role).
- `GET /api/validate/stats`: pass/fail/warn stats and recent logs.
- `GET /api/validate/gates`: gate metadata.

OpenAPI document: `docs/openapi/validate.openapi.json` (generated by `npm run docs:openapi`).

## CLI
Use the same core validator from terminal:

```bash
npm run validate -- --dir data/capsules --strict --report
npm run validate -- data/capsules/capsule.foundation.capsuleos.v1.json --fix
```

Supported flags:

- `--dir <path>`
- `--fix`
- `--watch`
- `--strict`
- `--skip-g16`
- `--ids-file <path>`
- `--output <path>`
- `--report`
- `--format <json|pretty|html>`
- `-v, --verbose`
- `--remote <url>`

## Background Audit
Run periodic vault audits:

```bash
npm run audit:capsules -- --fix
```

The audit script revalidates all capsules, optionally auto-fixes trivial issues, appends audit logs, and raises non-zero exit codes for critical failures (G04/G16).

## Pre-commit and CI

- Pre-commit hook (`.husky/pre-commit`) executes `npm run validate-staged` on staged capsule JSON files.
- CI workflow (`.github/workflows/validate-capsules.yml`) runs tests and validates all capsules on push/PR.

## Extending the Validator
You can register plugins with custom validation behavior:

```ts
import { createCapsuleValidator } from '@/lib/validator';

const validator = createCapsuleValidator();
validator.registerPlugin({
  id: 'tilesims-geometry',
  validate(capsule) {
    // return { errors: [...], warnings: [...] }
  },
});
```

## Security Notes

- Validation endpoints require authorization token.
- Batch/fix endpoints require owner role semantics.
- In-memory rate limits are applied per request identity.
- Validation and audit logs store metadata and issues only; do not emit full payloads in error channels.
